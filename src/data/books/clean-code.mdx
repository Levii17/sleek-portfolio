Before reading this book most of what I knew about
"good code" was pattern-matched from other people's repos rather than
reasoned about directly. This book was one of the first places I saw the
reasoning written out explicitly, function length, naming, the cost of
comments that lie, the idea that code is read far more often than it's
written.

## What stuck

The chapters on naming and function size changed how I actually write code,
not just how I think about it. Small, well-named functions that do one
thing are easy to agree with in the abstract; actually restructuring a
messy function into three smaller ones, under a deadline, is a different
skill, and this book at least made the target clear.

## Where I push back

Some of the specific rules (function argument counts, the strict stance
against comments) read as more dogmatic than I'd apply them today. Real
codebases have constraints this book doesn't have to deal with. I'd treat
it as a strong default, not a rulebook to follow literally.

## Would I recommend it

Yes, especially early in a career, with the caveat that it's a starting
point for taste, not a finished one.
